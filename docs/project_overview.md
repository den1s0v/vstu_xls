# Обзор проекта

**vstu_xls** — инструмент для извлечения из полуструктурированных человеко-ориентированных таблиц Excel структурированные данные, пригодные для программной обработки.  
Полуструктурированность здесь понимается в том смысле, что информация о связях между элементами данных неявно заключена в их взаимном расположении (аналогия --- шахматы).

Рабочим примером таких документов выступает коллекция xlsx-файлов с расписанием занятий ВолгГТУ.

## Идея

Исходные файлы составляются вручную: вариативны формулировки, активно применяется объединение ячеек, визуальное оформление. Вместо жёсткого «один формат на весь университет» проект опирается на **настраиваемую грамматику**: описание ожидаемых фрагментов таблицы (области, ячейки, связи «где что лежит») и правил их сочетания.

Парсер сопоставляет таблицу с грамматикой и строит дерево совпадений (**matches**), из которого затем несложным преобразованием формируется "автономный" JSON для дальнейшей загрузки в БД просмотрщика расписаний.

## Ограничения

Фокус парсера — практические сценарии (например, очное расписание одного вуза -- много файлов с типовыми элементами структуры); не заявляется универсальный разбор любого Excel.
Смысл структуры конкретного вида формулируется человеком в виде грамматики в явном виде.
Никакого машинного обучения и эвристик для угадывания того, что "подразумевалось", не предусмотрено.

## Грамматика

_Аналогия_. Подход позаимствован из синтаксических анализаторов для языков программирования (первый этап работы компилятора/интерпретатора), где поток символов входного текста сначала дробится на "слова" или типизированные токены, осмысленные для языка (**лексический** анализ), а затем поток этих токенов подвергается **синтаксическому** анализу для группировки их в выражения, инструкции, директивы, функции, классы, модули и т.д.

В этом проекте реализована похожая схема:  
На вход подаётся двумерная **сетка** ячеек с возможностью объединения ячеек в стиле Excel (только прямоугольные объединения).
- На первом этапе определяются наиболее вероятные **типы ячеек** (на основании специальной конфигурации с использованием _регулярных выражений_) -- аналог лексического анализа.
- На втором этапе работает парсер на основании заданных грамматикой **двумерных шаблонов** (паттернов), который группирует ("м*а*тчит") наборы найденных на первом этапе ячеек в более крупные образования, затем еще раз, и так, пока не придет к корневому паттерну, который описывает весь документ -- аналог синтаксического анализа.

Паттерны делятся на простые ("терминалы") -- просто ячейки на "поле", и составные ("нетерминалы").  
**Составные паттерны** ссылаются на другие паттерны, опционально задавая их **расположение** относительно себя. Если комбинация дочерних паттернов (с учетом всех заданных ограничений на позиции и размеры) найдена на поле, то паттерн считается найденным, и фиксируется совпадение (**матч**) для него.

Правильно определенные типы ячеек и паттерны --- залог корректной обработки документа парсером.
Наглядные **примеры** отлаженных грамматик можно найти в `cnf` (с комментариями) и `tests/test_data`.

> В дальнейшем будет добавлено описание логики работы парсера в текущей версии, так как это существенно для правильного подхода к составлению новых грамматик.


## Отладка

- Подробная **отладка** грамматики при выполнении парсинга (снимки по "волнам", размеченные совпадениями копии Excel-документа) включается отдельно и заметно увеличивает время работы.
- По умолчанию активирована **лёгкая диагностика** — файл `parsing_diagnostics.json` по каждому обработанному документу (см. [Диагностика парсинга](diagnostics_overview.md)).

## Основные подсистемы

- **Грамматика** — YAML-описание паттернов и типов ячеек (`grammar2d` и `string_matching`).
- **Сетка ячеек документа** — представление исходных данных (`grid`, `converters`).
- **Сервис разбора** — высокоуровневый запуск матчинга и экспорта отчётов (`document_parser`).

Более детальные подробности задокументированы непосредственно в исходном коде.
