# Диагностика процесса/результатов парсинга

## Зачем это нужно

Консольный вывод и отладочные сообщения плохо подходят для **пакетной** обработки: их нельзя автоматически собрать по каждому файлу. Структурированный отчёт **`parsing_diagnostics.json`** фиксирует предупреждения и ошибки за один прогон `parse_document` в одном месте на диске.

## Где лежит файл

- В демо **batch**: в той же подпапке отчёта, что и `unused_patterns.json` (имя подпапки обычно совпадает с именем исходного `.xlsx`).
- В демо **debug**: в каталоге `--output` рядом с файлами волн.

Отключить запись: флаг `--no-diagnostics` в соответствующем скрипте или не задавать `diagnostics_output_dir` у `DocumentParsingService`.

## Что внутри файла

- **`document`** — метаданные прогона: пути к источнику и грамматике (если переданы), время начала/окончания, длительность в мс.
- **`summary`** — счётчики по уровням (`errors_count`, `warnings_count`, `infos_count`) и число корневых совпадений (`root_matches_count`).
- **`issues`** — список записей с полями `severity`, `code`, `message` и при необходимости контекстом (`pattern_name`, `wave_index`, `expected`, `actual`, `source`, `exception_type`, `detail`).

Подробный перечень полей и кодов — в [Формате отчёта диагностики](parser_diagnostics.md).

## Связь с тяжёлой отладкой

**Debug waves** (JSON/Excel по волнам) — отдельный режим: подробно показывает состояние матчинга, но дорог по времени. **`parsing_diagnostics.json`** — лёгкий обязательный слой для оперативного контроля качества разбора; его можно включать всегда при пакетной обработке.
